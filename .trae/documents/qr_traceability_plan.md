# 农产品二维码溯源系统设计方案

## 1. 核心设计思路与整体框架

### 1.1 系统架构
- **生成阶段**：管理员在后台为每一批（或每个独立包装）产品创建档案，系统根据"产品编码+批次/唯一ID"生成唯一访问URL，并编码为二维码图片。
- **展示阶段**：消费者扫码后，前端页面通过URL中的唯一标识请求后端API，获取产品详情并渲染页面。
- **交互阶段**：前端提供点赞、转发、收藏功能，后端实现防刷点赞机制。

### 1.2 数据流程
1. 管理员录入产品信息（品种、定植信息、图片视频、采收质量等）
2. 系统生成唯一标识并创建二维码
3. 消费者扫码访问对应产品页面
4. 后端根据唯一标识返回产品数据
5. 前端展示产品信息并处理用户交互
6. 后端记录用户交互数据并进行防刷校验

## 2. 前后端语言工具与技术栈

### 2.1 后端
- **语言**：Golang（已确认）
- **框架**：推荐使用 Gin，理由：
  - 轻量级高性能，适合API服务
  - 丰富的中间件生态
  - 简洁的路由定义
  - 良好的错误处理机制
- **数据库**：推荐使用 PostgreSQL，理由：
  - 强大的关系型数据库功能
  - 支持JSON数据类型，适合存储复杂结构
  - 事务支持完善
  - 良好的扩展性
- **缓存**：使用 Redis，用于：
  - 缓存热点产品数据
  - 实现防刷机制（存储用户IP、设备标识等）
  - 提高系统响应速度

### 2.2 前端
- **框架**：推荐使用 Vue.js + Vite，理由：
  - 轻量级，适合移动端H5场景
  - 响应式设计，适配不同设备
  - 组件化开发，提高代码复用性
  - 良好的性能表现
- **UI库**：推荐使用 Vant，专为移动端设计的组件库
- **二维码扫描**：使用 qrcode.js 生成二维码，使用浏览器内置扫码API或第三方库处理扫码

### 2.3 存储方案
- **图片/视频存储**：推荐使用对象存储服务，如：
  - 阿里云OSS
  - 腾讯云COS
  - AWS S3
  理由：
  - 高可靠性和可扩展性
  - 支持CDN加速，提高访问速度
  - 按需付费，成本可控

### 2.4 二维码生成
- **库**：使用 golang-qrcode 库在后端生成二维码
- **URL结构**：`https://example.com/product/{unique_id}`，其中 unique_id 由产品编码和批次/唯一ID组合生成

### 2.5 防刷机制
- **实现方式**：
  - 使用 Redis 存储用户交互记录（IP、设备标识、时间戳）
  - 设置时间窗口限制（如5分钟内最多3次点赞）
  - 对异常行为进行监控和限制

## 3. 系统功能模块

### 3.1 后台管理模块
- 产品信息管理（CRUD操作）
- 二维码生成与管理
- 数据统计与分析
- 用户交互数据查看

### 3.2 前端展示模块
- 产品详情页（包含4个部分）
  - 产品基本信息（品种名、定植地点、定植时间）
  - 产品图片和视频
  - 采收质量信息
  - 用户交互反馈（点赞、转发、收藏）
- 响应式设计，适配移动端

### 3.3 后端API模块
- 产品信息查询API
- 用户交互记录API
- 防刷校验API
- 数据统计API

## 4. 数据库设计

### 4.1 核心表结构
- **products**：存储产品基本信息
- **product_batches**：存储批次信息
- **media_files**：存储图片视频信息
- **harvest_quality**：存储采收质量信息
- **user_interactions**：存储用户交互数据

### 4.2 数据关联
- 产品与批次为一对多关系
- 批次与媒体文件为一对多关系
- 批次与采收质量为一对一关系
- 批次与用户交互为一对多关系

## 5. 实现步骤

1. 搭建后端基础框架（Gin + PostgreSQL + Redis）
2. 设计并实现数据库表结构
3. 开发后台管理系统API
4. 实现二维码生成功能
5. 开发前端产品详情页
6. 实现用户交互功能和防刷机制
7. 测试与优化
8. 部署上线

## 6. 风险与应对措施

### 6.1 潜在风险
- 二维码被恶意复制或篡改
- 大量并发访问导致系统压力
- 防刷机制可能影响正常用户体验
- 图片视频存储成本控制

### 6.2 应对措施
- 为二维码添加防伪标识
- 优化系统架构，增加负载均衡
- 调整防刷策略，平衡安全性和用户体验
- 合理设置存储策略，定期清理无用数据
